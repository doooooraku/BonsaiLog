## 目的

14 作業の **「入力フィールド → HistoryTab チップ → PDF 表示 → CSV 列」** の対応を明文化し、入出力スキーマの整合性を保証する。

`principles.md` v1.5「入出力スキーマ整合性」チェック項目の参照ドキュメント。新しい入力フィールドを足すとき／既存フィールドの key を変えるときは、本ファイルを必ず更新する。

---

## 背景

v1.8 議論で判明した問題:

- `wireframes/care-screens-v2.jsx` は 14 作業 × 平均 2-3 項目 = **約 30 種の構造化フィールド**を生成している
- 一方 `wireframes/detail-screens.jsx` の `HistoryTab` は **`note` 1 個と `photos` 枚数しか表示せず**、構造化情報をほぼ全消失
- `MOCK_HISTORY` で `note: '2mm 幹に / 12週後に外し予定'` のように **手書きで構造を再構築**する状態だった
- PDF レポート (`export-screens.jsx:786`) と CSV (`export-screens.jsx:547`) の粒度も別々に進化していた

→ **入力スキーマと表示スキーマの対応表が無い**ことが根本原因。本仕様書がその不在を埋める。

---

## 共通規則

1. **チップラベルは `_Seg` のラベル文字列を再利用**する（例: `'normal'` → 「いつも通り」）。バックエンド key は内部値、UI 表示は人間語ラベル。両者の対応は `_LABEL` マップ一箇所で管理する。
2. **チップ順序は入力フォームの上から下**に揃える。フォームで「番手 → 部位 → 外し予定日」の順に書いた以上、HistoryTab でも同順で並ぶ。
3. **長文の自由テキスト**（用土レシピ・薬剤名・処置内容・銘柄）はチップとして並べる。`max-width` で ellipsis 切り捨て。
4. **note は自由メモ専用**。チップで選んだ情報を note に書き直す必要は無い旨をフォームの placeholder で示唆済（`care-screens-v2.jsx:469` 「自由メモ」）。
5. **写真サムネは別行**（最大 3 枚 + `+N` バッジで省略）。
6. **14 作業すべて同じ `_HistoryChip` / `_ChipRow` コンポーネント**を再利用し、見た目の揺れを排除する。
7. **PDF/CSV エクスポート**との粒度整合は、本仕様書の各作業セクション「PDF 表記」「CSV 列マッピング」で明示する。

## 14 作業の表示テンプレート

各作業セクションは以下の 4 段で記述する:

- **入力**: `ActionFormFields` (`care-screens-v2.jsx:80-300`) で生成される state キー
- **HistoryTab チップ**: `buildChipsFor(entry)` が返すチップ配列
- **PDF 表記**: `export-screens.jsx:786` で個別 PDF に書き出す行
- **CSV 列**: `export-screens.jsx:547` (`event_id, ..., part, amount, dilution, duration, memo, photo_count, ...`) のどの列にどう詰めるか

---

### 1. 水やり (`water`)

| 段     | 値                                                                                            |
| ------ | --------------------------------------------------------------------------------------------- |
| 入力   | `amount`: `normal` / `plenty` / `light`                                                       |
| チップ | **0〜1 個**（`normal` は冗長なので省略、`plenty`/`light` のみ強調） — `[たっぷり]` / `[軽く]` |
| PDF    | 「水やり: たっぷり」（normal は省略可）                                                       |
| CSV    | `amount = normal/plenty/light`（CSV は省略しない）                                            |

**根拠**: 水やりは記録総数の大半を占め、`normal` がデフォルト。すべての水やり行に `[いつも通り]` チップが並ぶと range 展開時に視覚ノイズになる。`plenty`/`light` のレアケースだけ強調する方がスキャン性が上がる。CSV では機械可読性のため省略しない。

### 2. 剪定 (`prune`)

| 段     | 値                                                                |
| ------ | ----------------------------------------------------------------- |
| 入力   | `parts[]`: `eda`/`ha`/`shinme`/`ne`、`amount`: `tip`/`mid`/`bold` |
| チップ | 部位 N 個 + 量 1 個 — 例 `[枝]` `[葉]` `[そこそこ]`               |
| PDF    | 「剪定: 枝・葉 / そこそこ」                                       |
| CSV    | `part = eda+ha`, `amount = mid`                                   |

### 3. 針金がけ (`wire`)

| 入力   | `gauge`、`parts[]`: `all`/`miki`/`eda`、`removeDate`: ISO 日付     |

| PDF    | 「針金がけ: 2mm / 幹 / 外し 2026-06-02」                           |

## 写真サムネ仕様

- **配置**: 構造化チップ行と note 行の **下** に独立した横並び
- **サイズ**: 88×88、`borderRadius: 8`
- **最大表示数**: **3 枚** + `+N` バッジ（4 枚以上は最後を「+(N-3)」バッジで集約）
- **既定**: ON（プライバシー OFF オプションは v2 で検討）
- **タップ挙動**: v1 は無し（モーダル展開は v2）

## ラベル辞書 `_LABEL`

実装側 (`detail-screens.jsx`) では以下のフラットなマップを定義する。`buildChipsFor` から参照する単一情報源。

```js
const _LABEL = {
  // amount (water)
  normal: 'いつも通り',
  plenty: 'たっぷり',
  light: '軽く',
  // amount/scope (prune/leaf/defol/bud/mekiri)
  tip: '枝先のみ',
  mid: 'そこそこ',
  bold: '思い切り',
  // parts (prune)
  eda: '枝',
  ha: '葉',
  shinme: '新芽',
  ne: '根',
  // parts (wire/unwire)
  all: 'すべて',
  miki: '幹',
  // kind (fert)
  solid: '置肥（玉肥）',
  liquid: '液肥',
  slow: '緩効性',
  other: 'その他',
  // purpose (spray)
  prevent: '予防',
  treat: '治療',
  both: '両方',
  // symptoms (heal)
  yake: '葉焼け',
  kare: '枝枯れ',
  mushi: '虫',
  kabi: 'カビ',
  // tasks (moss)
  add: '貼り直し',
  remove: '剥がす',
  water: '湿らす',
};
const _ROOTS_LABEL = { none: '整理なし', light: '軽く整理', half: '1/3整理', heavy: '1/2整理' };
```

`heal.symptoms` の `'other'` は `_LABEL.other`（「その他」）と key が衝突するため、`'症状その他'` のような **コンテクスト付きラベル**で表示する。実装では `buildChipsFor` 内で symptoms の other だけ別ラベルに振り直す。

## PDF 個別レポート — メタ情報マッピング

PDF 個別レポートは **「展示会出品票・譲渡時の継承書類」** の用途で出力される **人間用ドキュメント**。CSV（機械可読）と異なり、**受け手（弟・買い手・展示会審査員）にとって意味のある項目のみ** を載せる。

### メタ行の項目選定（v1.10 で確定）

`wireframes/export-screens.jsx:746-751` の `_PDFMeta` 行に表示する項目：

| 順  | 項目         | 出典（DB）             | 採否                     | 理由                                                                  |
| --- | ------------ | ---------------------- | ------------------------ | --------------------------------------------------------------------- |
| 1   | **銘**       | `bonsai.gou`（任意）   | 採用（任意・あれば表示） | 展示会の銘板・継承書類の正統な項目。命名銘がない場合は **行ごと省略** |
| 2   | **入手日**   | `bonsai.acquired_at`   | 採用                     | 由来を示す事実                                                        |
| 3   | **入手元**   | `bonsai.acquired_from` | 採用                     | 由来・血統を示す事実                                                  |
| 4   | **樹齢推定** | `bonsai.age_estimate`  | 採用                     | 価値判断の参照値                                                      |
| 5   | **樹高**     | `bonsai.height_cm`     | 採用                     | サイズ把握の事実                                                      |
| 6   | **鉢サイズ** | `bonsai.pot_spec`      | 採用                     | サイズ把握の事実                                                      |

### 内部 UI 専用ラベル — PDF 個別レポート不掲載リスト（v1.10 で確定）

以下は **CSV には載せるが PDF（人間用）には載せない**：

| 項目                               | CSV      | PDF      | 不掲載の理由                                                                               |
| ---------------------------------- | -------- | -------- | ------------------------------------------------------------------------------------------ |
| **ID**（`bonsai.id`）              | 載せる   | 載せない | 内部識別子。受け手に意味不明（`abc-123` は弟・買い手・審査員に読めない）                   |
| **タグ**（`bonsai.tags[]`）        | 載せる   | 載せない | 内部フィルタ用ラベル（例 `#展示会候補`）。所有者の作業フラグであり外部公開を前提としない   |
| **置き場所**（`bonsai.location`）  | 載せる   | 載せない | 所有者の運用情報。譲渡先・展示会場では情報価値ゼロ。SNS 拡散時の **位置情報リーク** リスク |
| **飼育年数**（入手日からの経過年） | 計算可能 | 載せない | 樹齢（例 35 年）と混同される。同行併記は読み手を混乱させる                                 |

### 任意項目の fallback

「銘」のように **任意項目** は値が無いとき **行ごと省略** する。空文字列 `""` のまま PDF に出さない。

```jsx
{
  bonsai.gou && <_PDFMeta k="銘" v={bonsai.gou} />;
}
```

### CSV と PDF の役割分担

| 観点             | CSV (`bonsai_csv` 第 1 列〜)   | PDF（個別レポート）                    |
| ---------------- | ------------------------------ | -------------------------------------- |
| 想定読者         | 機械（インポート・データ移行） | 人間（弟・買い手・展示会審査員）       |
| ID               | 必須                           | 不要                                   |
| 内部運用情報     | 含む（`location` `tags`）      | 除外                                   |
| プライバシー配慮 | 出力者の責任                   | UI 側で内部ラベル混入を防ぐ            |
| 命名銘           | 任意                           | 任意（あれば表示・なければ行ごと省略） |

→ **CSV は機械可読・PDF は人間用** が大原則。同じデータでも出力先で項目を切り分ける。

### 写真選定

PDF 左上のカバー写真は v1 では `seed={0}`（最初の写真）固定。v2 で「カバー写真選定」（最新／ユーザー選択／ベスト判定）を検討（実装スコープ外）。

### v1 → v2 で送る PDF 関連項目

- 用途別レポート切り替え（展示会／継承／自家保管 のセグメント追加）
- カバー写真選定 UI
- タグの `private/public` フラグ（公開可能なタグだけ PDF に出す）

### 一覧 PDF 不掲載リスト（v1.11 で確定）

| 項目                                | 旧（v1.10） | 新（v1.11） | 不掲載の理由                                                                                                                                                                         |
| ----------------------------------- | ----------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **最終水やり**（「今日」「2日前」） | 載せていた  | **削除**    | 印刷物（時間停止）に動的相対時刻が混入。生活パターン推測のプライバシーリスク。「水やり当番チェック」用途は画面（Home）でリアルタイム確認すべきタスクで、PDF にする時点で意味が消える |

### 表紙ページ（`PDFListCover`）

維持項目: 発行日 / 記録期間 / 盆栽総数（アーカイブ除く） / 樹種 / 記録総数（5 行）。

**MOCK 整合ルール**: 表紙の「樹種 N 種」「盆栽総数 N 本（アーカイブ除く）」と一覧ページ `items[]` の樹種数・本数は必ず一致させる。`PDFListStatsPage` の樹種別保有数も同源データで集計される前提で MOCK を作成する。

### 統計ページ（`PDFListStatsPage`）

維持項目: 6 数値カード（記録総数・水やり・剪定・施肥・植替え・消毒、過去 1 年） / 樹種別保有数 / 月別記録数（過去 12 か月）。

**v2 候補**: ExportOptionsSheet の期間設定（v1.4 実装済）に統計ページを連動させる（現状は「過去 1 年」固定）。`include_stats: bool` オプションで optional 化も検討。

## PDF 出力ルール — 時間表記正規化（v1.11 で確定）

PDF（印刷物）は **時間が停止した出力物** であり、画面 UI の動的相対時刻と特性が異なる。生成時点で時間表記を正規化する。

### 正規化ルール

| 出力先              | 推奨表記                                               | 禁止表記                             | 理由                                                                 |
| ------------------- | ------------------------------------------------------ | ------------------------------------ | -------------------------------------------------------------------- |
| **PDF（印刷物）**   | ISO 日付（`2026-04-24`）または絶対表記（`過去 90 日`） | 「今日」「昨日」「2 日前」「N 日前」 | 印刷後・PDF 保管後に意味が変わる。受け手が「いつの今日？」と困惑する |
| **CSV（機械可読）** | ISO 日付タイムスタンプ（`created_at` 列）              | 相対時刻文字列                       | 機械可読性のため相対時刻は不可                                       |
| **画面 UI**         | 「今日」「2 日前」（相対時刻 OK）                      | ─                                    | リアルタイム性が前提。スキャン性のため相対時刻が望ましい             |

### PDF 全般の時間表記方針

- **個別 PDF メタ行**: `入手日 2020-03-15`（ISO 日付）
- **個別 PDF 作業履歴サマリ**: 「過去 90 日」（期間ラベル）+ 個別行は ISO 日付（例 `2026-04-22`）
- **一覧 PDF 表紙**: 「発行日 2026-04-24」「記録期間 2017-09-30 〜 2026-04-24」（ISO 日付）
- **統計ページ**: 「過去 1 年」「2025 年 5 月〜2026 年 4 月」（期間ラベル）

### 例外

「飼育 6 年」のような **期間表現**（経過時間）は相対時刻ではなく **計算済み事実** のため PDF で OK。動的に変化するが、生成時点でスナップショットされる。

## 個別 PDF ↔ 一覧 PDF 項目整合表（v1.11 で確定）

PDF 出力物（個別 / 一覧 / CSV）の項目選定が独立進化しないよう、整合状態を明文化する。

### 整合表

| 項目       | 個別 PDF（v1.10） | 一覧 PDF（v1.11）                       | CSV                      | 整合方針                                     |
| ---------- | ----------------- | --------------------------------------- | ------------------------ | -------------------------------------------- |
| ID         | ✗                 | ✗（連番のみ）                           | ◯                        | CSV のみ                                     |
| 銘         | ◯（任意・行省略） | ✗（v2 候補）                            | ◯                        | v2 で一覧にも検討                            |
| 入手日     | ◯（ISO）          | ◯（ISO）                                | ◯                        | 完全整合                                     |
| 入手元     | ◯                 | ✗（一覧では省略合理）                   | ◯                        | 用途差で許容                                 |
| 樹齢推定   | ◯                 | ◯                                       | ◯                        | 完全整合                                     |
| 樹高       | ◯                 | ◯                                       | ◯                        | 完全整合                                     |
| 鉢サイズ   | ◯                 | ✗（4 列維持で省略）                     | ◯                        | 用途差で許容（v2 検討）                      |
| 飼育年数   | ✗（樹齢と混同）   | ◯（一覧では別行で並ぶため混同リスク低） | ◯                        | 文脈差で扱い分け（明示済）                   |
| 置き場所   | ✗（プライバシー） | ✗                                       | ◯                        | 完全整合                                     |
| タグ       | ✗（内部 UI）      | ✗                                       | ◯                        | 完全整合                                     |
| 最終水やり | ✗                 | ✗（v1.11 で削除）                       | ◯                        | 完全整合（CSV は `events_csv` で時系列保持） |
| 記録総数   | ✗                 | ✗（v1.11 で削除）                       | ─（events_csv で集計可） | 完全整合                                     |

### 整合のチェックポイント

新規項目を PDF（個別 or 一覧）に追加する／既存項目の表記を変えるとき：

1. 個別 PDF と一覧 PDF の **両方で同じ項目を扱うか** 検討する
2. 用途差（個別＝展示会出品票・継承書類 / 一覧＝保有資産一覧）で **採否を分ける合理性** があれば本表に明記
3. **動的相対時刻**（`今日`・`N 日前`）が混入していないか「PDF 出力ルール — 時間表記正規化」で照合
4. **MOCK の樹種数・本数・統計値の整合性**（表紙 ↔ 一覧 ↔ 統計ページ）を維持

## v1 → v2 で送る項目

実装スコープ外として記録:

1. **CSV 列の構造化拡張**: `part` `amount` を JSON 形式 or 列分離して、構造化情報の機械可読性を上げる
2. **写真モーダル全件ビュー**: タップで全件サムネ + 拡大プレビュー
3. **設定で「Home でメモを表示」OFF**: プライバシー閾値（自由テキストに個人名混入リスク）
4. **同じ作業タイプの比較ビュー**（縦串）: 過去の植替え 3 回を並べる視点（プロ層向け）
5. **多言語化** (i18next 化, P3-1): `_LABEL` のキー → 訳語マッピング 19 言語
6. **MOCK_HISTORY 拡張**: `heal` / `moss` / `move` / `leaf` / `defol` / `bud` / `unwire` の MOCK 例を追加（同時にフィルタチップ拡張・アイコン統一）。v1.5 では `labelFor` のみ 14 作業対応済、MOCK と icon は未対応のため見た目検証は v2 で。
7. **一覧 PDF 写真大型版オプション**（v1.11 で確定）: 1 ページ 5 本 → 1 ページ 3 本へ。みどり（家族提示）／松本（SNS 元ネタ）向け。`include_large_photo: bool` で切替。
8. **一覧 PDF 用途別レポート**（v1.11 で確定）: 「保有資産一覧／棚管理／譲渡準備」の 3 種類セグメント。ExportOptionsSheet に「用途」セレクタ追加。
9. **健康度ダッシュボード**（v1.11 で確定）: 「最終水やり当番チェック」用途は **画面（Home）でリアルタイム確認すべきタスク** のため PDF ではなく画面 UI で対応。
10. **名前と銘の統合表示**（v1.11 で確定）: 一覧 PDF の名前列を「父の黒松 — 結松」のように統合表示するか検討。4 列維持の制約と両立可能か。
11. **連番ラベルの再設計**（v1.11 で確定）: `#1`〜`#N` の表記が「リスト順序」か「内部 ID」か曖昧。並び順指定 UI と連動させるか、サムネのみで識別する設計に変更するか。
12. **統計ページ optional 化**（v1.11 で確定）: ExportOptionsSheet で `include_stats: bool` を追加。期間設定（v1.4 実装済）に統計ページを連動させ、固定「過去 1 年」を撤廃。
13. **アーカイブ盆栽を含めた一覧出力**（v1.11 で確定）: 表紙「盆栽総数 5 本（アーカイブ除く）」をオプションで「アーカイブを含む」に切替できる出力モード。
14. **Onboarding 引き継ぎ分岐**（v1.13 で確定）: 初回起動時の Onboarding に「以前の端末から引き継ぎ（ZIP インポート）」分岐を追加。設定画面まで掘らないと到達できない現状を解消。みどり（家族 iPad 機種変更）／高橋茂（業務端末更新）の動線改善。
15. **定期バックアップ提案 UI**（v1.13 で確定）: 設定画面または BackupScreen に「最後にバックアップした日」表示と「定期バックアップ提案」ヒント。命令ではなく「事実の表示＋ユーザー判断」哲学に揃える（「○日以上バックアップしていません」のような提示はしない／「最終バックアップ 2026-04-22」のみ）。
16. **ZIP スキーマバージョン仕様**（v1.13 で確定）: バックアップ ZIP の中身仕様（写真込み・JSON フォーマット・スキーマバージョン）を `display-schema.md` に明文化。後方互換性ルール（v1 で書き出した ZIP は v2 で取り込めるか）を含む。
17. **iCloud バックアップとの役割分担**（v1.13 で確定）: iOS 標準 iCloud バックアップは「同一 Apple ID 端末間の自動復元」、アプリ内 ZIP は「Android／PC 移行・アプリ削除前保全・家族共有 iPad の引き継ぎ」用途。両者の補完関係を `display-schema.md` または `principles.md` に明文化。
