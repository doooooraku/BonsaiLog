はじめに ― この文書の読み方

この文書は「BonsaiLog ワイヤーフレーム v0.1」の UX レビュー記録 です。新機能を作るのではなく、すでにある画面を 「使う人の視点」で点検 し、改善点を整理することが目的です。

BonsaiLog は盆栽愛好家のためのスマホアプリ ― 盆栽を登録し、毎日の作業（水やり・剪定・植替えなど）を記録し、写真を貯めていくサービスです。すでに 35 画面・5 機能領域 のワイヤーが用意されており、デザインの完成度は高い水準にあります。

今回はその 完成度を疑う側 に立って、「初めて触る人」「シニア」「プロ」「英語ユーザー」など、立場の違う6人を仮想的に登場させ、画面の中で実際に詰まる/迷う/離脱する箇所を洗い出しました。さらに、6人の専門家（UX デザイナー、盆栽家、PM、アクセシビリティ、i18n、行動心理）に独立してコメントしてもらい、判断のバイアスを避けています。

読む順番のおすすめ
  1. 急ぎなら TL;DR と 課題一覧 だけ。
  2. 10分あるなら + 画面遷移図 v1.5（別ファイルに統合）。アプリ全体の地図がわかります。
  3. 30分あるなら + 動線シミュレーション。ユーザーが「どこで詰まるか」が体感できます。
  4. 意思決定者なら + 改善ロードマップ。何を、いつ、どう直すかの提案です。

用語が出るたびに 巻末の用語集 で説明していますので、専門用語に詰まったらそちらをご覧ください。

検証方法 ― 多視点シミュレーション

バイアスを排除するため、ユーザー側 6 人 × 専門家側 6 人 × 動線 5 つを掛け合わせ、計 180 通りの視点でワイヤーを点検しました。

① ペルソナ生成（6名）

盆栽愛好家層を 「経験」「年齢」「動機」「環境」 の4軸で広げ、抜けがないように配置。極端なペルソナ（初心者すぎる・プロすぎる・英語のみ）を意図的に含めることで、「まんなか向け」だけにならないよう設計。

② 専門家パネル（6視点）

UX、盆栽、PM、アクセシビリティ、i18n、行動心理。各専門家は 独立にコメント し、お互いの結論に依存させない（情報カスケードを避ける）。最後に合議で優先度を決定。

③ 動線シミュレーション（5動線）

「初回登録」「日々の水やり」「週末まとめ作業」「振り返り」「データ持出し」の5フローを、各ペルソナが歩く想定で ステップ単位 で追跡。詰まる場所と離脱しそうな場所を記録。

④ 仮説検証ループ

1周目で見つけた問題を仮説として立て、2周目でそれが本当に各ペルソナで起こるか再検証。3周目で対応案を逆シミュレーション（直したら誰が幸せになるか／誰が困るか）まで実施。

バイアス対策
「私たちが作ったから直さない」という結論を避けるため、3周目では 悪魔の代弁者（devil's advocate） ロールを置き、「これは現状維持で正しいのではないか？」を主張させました。最終的な P0 5件は、悪魔の代弁者が反論しきれなかった項目だけが残っています。

1 次データで言える4つのこと

差別化のチャンス（unowned moat）
調査した 7 アプリすべてで「一括記録」が欠落、そして GitHub-style 水やりヒートマップは1社も実装していない。BonsaiLog が両方を持っていることは 市場で唯一。ただし P0-1（一括選択の発見性）が解決されない限り、この差別化は実現していないのと同じ。発見性の改善は 機能改善 ではなく 差別化資産の解放 という位置づけ。
市場が嫌う声（避けるべき道）
Planta（「頻度が外れる」「情報が間違い」）、Greg（「サブスクの裏に通知を隠していてずるい」）、PictureThis（「Onboarding の課金が攻撃的」）― 共通するのは 「アプリが正しいと言い張る」 姿勢への嫌悪。BonsaiLog の 「目安」「参考」しか使わない 言葉遣い、「これは記録の表示です。水やりの判定はしません」 という明示は、これらの轍を回避する正しい立ち位置。絶対に変えるべきでない。
日本のシニアスマホ事情（総務省・NICT 1次統計）
60 代の iPhone/Android 所有率：77.3%、70 代：60.6%（NICT、総務省 令和3年版情報通信白書）。70+ で「使い方がわからない」 が 42.4%。これは 佐藤みどり（62）の摩擦体験は誇張ではない、ど真ん中の現実であることを意味する。
一方、日本盆栽協会の会員数は 1980 年の23,000人 → 現在 約3,000人 に縮小（日本盆栽協会）。中心層の高齢化は事実、若年化／女性化は マスメディアの主張があるだけで統計裏付けは薄い。シニアファースト UX を捨てるべきではない。
「素早く記録できる」の業界標準
MyFitnessPal は 「記録時間を数分から数秒へ」を明示的設計目標として宣言（公式）。ウィジェット／ロック画面からのクイックログで 28〜62% 時間短縮。Day One も 「数秒で1エントリ」を売り（公式）。
BonsaiLog の現状は カードタップ → ＋ → Sheet → 入力 → 保存 の 4〜6タップ。サブ15秒は達成可能だが、サブ5秒には iOS ウィジェット／ロック画面ショートカットが要る。「水やり」だけはそこを目指したい。

市場リサーチで 新たに浮上 した懸念

  1. 16作業 × デフォルト ON 通知 = 通知疲れ。Greg/Planta が叩かれた最大の要因。BonsaiLog は通知を 作業種別ごとに個別 ON し、初期は OFF が安全。設定画面の「通知の時間帯（8:00–20:00）」だけでは粒度不足。P1 として追加。
  2. サブ5秒の水やり記録。iOS ウィジェット または Lock Screen Live Activity で、「父の黒松に水やり」 をワンタップで登録できる経路があれば、Planta/Greg/Bonsai Album いずれにもない強烈な差別化に。P3 として観察。
  3. Free 5本制限の妥当性。Bonsai Care App は Free 5本まで。BonsaiLog は登録数 ∞ にしているのは 正しい判断。価格帯の比較で買切¥9,800 は競合より高めだが、「サブスクではない」がプロ層に刺さる。継続観察。
  4. Mirai Mobile の「樹種＋地域別カレンダー」は強烈な診断系価値。BonsaiLog は 診断しない 立ち位置を堅持しつつ、「地域別の 参考 カレンダー」（オプション）を Pro 機能として用意するのは検討余地あり。P3。

専門家パネル ― 6視点の独立所見

同じワイヤーを6人がそれぞれ別室で評価し、最後に持ち寄ったコメント。意見が割れた箇所はそのまま残してあります。

UX デザイナー

動線設計は 記録系アプリの相場以上。Home → 詳細 → 作業選択 Sheet → フォーム の階層が深すぎず、Sheet 内のワンタップ確定（ピッカー）が良い。最大の懸念は発見性：長押し1択の一括選択、Tab バーの「記録」ラベル（具体的に何が入るか不明）、Mermaid 図にあるが JSX 未実装の 「・・・」メニュー。「設計はあるが、ユーザーは見つけられない」状態。

盆栽家

樹種ピッカー9種、樹形10種は 初心者から中級までカバー。学名併記 ＋ カスタム入力で、マイナーな樹種にも逃げ場あり。気になるのは (1) 「目安」の表示位置と程度（ suggestion: trueのフラグが timeline mock に入っていて、「春の施肥目安」 のように表示される。プロは嫌うが初心者は欲しい）、(2) 「葉刈り」と「全葉刈り」の同一フォーム（後者は禁忌に近い場面がある、せめて季節警告）、(3) 用土の「赤玉土:桐生砂 = 7:3」表記は素晴らしい、自由記述で残してほしい。

プロダクトマネージャ

Pro 訴求の入り方が 機能ロックの暴力に近い。Free と Pro の境界線：「写真 1鉢3枚まで」、「CSV/PDF 全 PRO」、「年次タイムライン PRO」。
Apple 規約 3.1.1 の「購入を復元」は paywall・設定の両方にあり、これは合格。Apple 規約 3.1.2「自動更新」明記もある。
提案：Free でも CSV 1種（最低限の盆栽一覧）は出せるようにし、「データを返さない」 印象を消す。差別化は「定期エクスポート（毎月の自動 CSV メール）」「年次タイムライン画像生成」「写真の長期アーカイブ閲覧」など 付加価値型 に寄せる。

アクセシビリティ専門家

tokens.css に 屋外モード（WCAG AAA 7:1）が定義され、prefers-reduced-motion 対応もある ― 仕組みは優秀。
ただし 運用と整合していない：
・--tap-min: 48px 宣言だが _Seg の最小タップ高さは 32–40dp。
・屋外モードは 手動切替のみ（明るさセンサ自動なし）。
・SVG ヒートマップの色（薄: #EDE7D8 / 中: #A7B79E / 濃: #1F3A2E）は色覚弱者の Deuteranopia でやや識別困難の懸念。
・aria-label は 「閉じる」 や 「戻る」 に付くが、「保存」 「タブ各種」などの主要ボタンに見当たらない箇所がある。
・FAB の aria-label="盆栽を追加 / 長押しで作業記録" は 長押しを言葉で説明している ― これは VoiceOver には届くが、晴眼者の発見性問題は残る。

i18n エンジニア

言語ピッカー19言語、tokens の --font-display-ja / --font-body-ja 構造は他言語混在に強い。が、コンテンツ層が 日本語ハードコード：screens.jsx の Welcome 文言、care-screens-v2.jsx の作業ラベル、樹形 desc など。
さらに 「松類限定」を speciesOnly:'pine' で判定 ― これは樹種コード体系を i18n とどう結ぶかが要設計。
提案：(a) i18n 層（i18next 等）を介した文言管理に移行、(b) 単位系（cm/inch、°C/°F）を端末ロケールから自動推定＋設定で上書き可、(c) 樹形・作業の 意味的英訳 を辞書化（Decandling, Defoliation 等）。

行動心理学者

習慣化の観点で GitHub-style ヒートマップは秀逸。連続記録（streak）を可視化することで、損失回避が働き、続ける動機を提供する。これは Apple Health のリングや GitHub の草と同系統。
ただし 初期不安（健太ペルソナ）への配慮が薄い：Onboarding 直後に「あなたの最初の30日」のような 進捗が見える小さな道標 がない。
提案：30日ガイドを Onboarding 完了後に 1つだけ提示し、毎日「今日のチェック」を1件だけ出す。これで「何をすべきかわかる安心」と「ヒートマップに最初の点が打てる達成感」が同時に手に入る。
※ v1.1 で 30日ガイド削除（哲学整合）／ v1.7 で 連続記録（streak）表示削除（Streak 文化は盆栽の水やり判断に逆効果。樹種別の乾湿差を無視した連続強化は過湿リスクを生むため）

  1. タブバーの「記録」ラベルが指す画面は？
    home-screens.jsx の TabBar 定義は「盆栽 / 予定 / 記録 / 設定」。「記録」 をタップすると何が出るのか、JSX では未実装。水やりヒートマップ？　履歴一覧？　検索？　― ユーザーは 記録 という単語に何を期待するのか、ヒアリングが必要。
  2. iCloud / Google Drive 同期は対応するのか？
    設定には「ZIP インポート」のみ。複数端末で使いたい人（鈴木大輔ペルソナ）にとっては必須に近い。リスクは（同期の競合）（プライバシー）（運用コスト）。同期しない＝端末紛失でデータ消失のリスクをどう説明するか。
  3. 松類限定の「芽切り」を、どう樹種拡張に対応させるか？
    speciesOnly:'pine' はハードコード。欅・もみじの「全葉刈り」、真柏の「ジン・シャリ加工」 なども樹種限定。樹種コード × 作業 のマトリクス管理に切り替える時期では？
  4. 通知の頻度上限とトーンは？
    設定に「通知の時間帯（8:00–20:00）」あり。ただし1日に何件まで送るか、季節作業の「時期幅」（5月中旬〜6月初）をどう通知するか不明。
  5. Pro 価格帯の妥当性
    月¥500 / 年¥3,980 / 買切¥9,800。日本の盆栽愛好家層に向けた値付けとしては「高すぎる」も「安すぎる」も判断材料が不足。本気度フィルターとして妥当か、ユーザー調査が要。
  6. 盆栽園プロ向けの「弟子・顧客共有」
    高橋茂の動線では「PDF出力 → メール／LINE」が現実解。ただし弟子3名のスマホで 同じデータベース を見たいというニーズは高そう。マルチユーザー機能は別アプリにするか、Pro Plus で出すか。

用語集 ― この文書で使った言葉

専門用語に詰まったら、ここを見てください。中学生にもわかる説明にしています。

FAB（ファブ）
Floating Action Button の略。画面の右下に浮かぶ円形ボタン。最も大事な操作（このアプリでは「盆栽を新しく登録する」）に使う。
モーダル
今の画面の上に重ねて出てくる、専用の画面。閉じるまで他のことができない。「新規登録」がそれ。
ボトムシート
画面の下から「ぴょこっ」と上がってくる小さな画面。下にスワイプすれば閉じられる。「樹種ピッカー」「作業選択」がそれ。
ハブ（中継点）
たくさんの画面が放射状につながっている中央。BonsaiLog では Home がハブ。
タップ領域
指で押せる範囲。Apple/Google ともに「48×48 ピクセル以上」を推奨。手袋／老眼／屋外で誤タップを減らすため。
セグメンテッドコントロール
「いつも通り / たっぷり / 軽く」のように、横に並んだ選択肢から1つ（or 複数）選ぶUI。
VoiceOver
iPhone のスクリーンリーダー。視覚に障害のある人に「画面に何が書いてあるか」を読み上げてくれる。aria-labelで何を読むか指定する。
WCAG / JIS X 8341
WCAG はウェブのアクセシビリティ国際規格。JIS X 8341 はその日本版。「文字とその後ろの色のコントラスト比7:1以上」など、誰でも見やすくするための基準。
i18n（インターナショナリゼーション）
「i」と「n」の間に18文字あるからこの略。多言語化の意味。文章だけでなく、日付の書き方、単位（cm/inch）、絵文字などすべてを「その国に合わせる」設計のこと。
ATT（App Tracking Transparency）
iOS 14.5 以降、広告のための端末ID（IDFA）使用には「許可しますか？」のダイアログが必須。Apple の規約。
UMP（User Messaging Platform）
Google が提供する、EU/UK/CH の GDPR 同意取得を簡単にする仕組み。広告事業者の使うデータの同意を取る。
Apple 規約 3.1.1
App Store のレビューガイドライン。「アプリ内課金で買えるものは、購入の復元（別端末で買い直さなくていい）を提供しなければならない」と定める。
P0 / P1 / P2 / P3
優先度。P0 = 直さないとリリースできない／致命的、P1 = 強く推奨、P2 = 改善余地、P3 = 観察項目。
FlashList
大量のリスト（150株とか）をスクロールしてもカクつかないようにする React Native のライブラリ。「60fps target」と書かれている。
GitHub-style ヒートマップ
GitHub のプロフィールにある「草」みたいな格子状の表示。色の濃さで「その日に何回やったか」を表す。BonsaiLog では水やりに使われている。
UTF-8 BOM 付き CSV
日本語版 Excel で文字化けしないために、ファイルの先頭に「これは UTF-8 です」というマーク（BOM）を付けた CSV のこと。技術者にとっては当たり前だが、これがないと多くのユーザーが困る。
樹形（じゅけい）
盆栽の幹の形の分類。直幹（まっすぐ）、模様木（くねっとした）、懸崖（崖から垂れる）など10種類くらいある。
芽切り（めきり）
松類で行う、新芽を一度切って二番芽を出させる技。葉のバランスを整える。専門技術。
針金がけ
盆栽の枝や幹に針金を巻いて、好きな形に曲げる作業。一定期間後に外す（食い込み防止）。
アーカイブ
このアプリでは「削除しないで隠す」こと。枯れた盆栽、譲った盆栽の記録は消さずにアーカイブして、いつでも復元できる。継承文化を尊重する設計。
habit formation（習慣形成）
「毎日続ける」を脳が苦にしなくなる仕組み作り。GitHub の草、Apple の達成リング、BonsaiLog の水やりヒートマップは皆この設計。
streak loss aversion
「連続記録が途切れるのが嫌」という心理。これを利用するのが habit formation の王道。BonsaiLog では v1.7 で 連続記録（streak）表示を削除。理由：盆栽の水やりは樹種別の乾湿差が必須で、連続強化は過湿リスクを生むため、哲学（命令しない・判定しない）と整合しない。
devil's advocate（悪魔の代弁者）
議論で「あえて反対意見を言う係」。集団で同じ結論に流れすぎないよう、わざと反論を出す役。
情報カスケード
先に発言した人の意見に、後の人が引きずられて同じ結論に流れていく現象。意見の独立性が崩れる。これを避けるため、本評議会では各専門家を別々に評価させ、最後に持ち寄っている。
groupthink（集団思考）
仲のいい集団内で「反対意見が出ない空気」が生まれ、結論が偏る現象。これを避けるため悪魔の代弁者を置く。
