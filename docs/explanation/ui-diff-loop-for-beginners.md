# 1 画面 UI 整合ループ — 初心者向けガイド

> **このドキュメントは「BonsaiLog の見た目を、 お手本通りに直す作業」 を、 小中学生にもわかるように説明します**。
> 既に慣れている人は `docs/how-to/ui-diff/screen-integration-loop.md` (技術者向け手順書) を直接見てください。
> 最終更新: 2026-05-17 (Sess5 PR-4 反映時点)

## 1. これは何の話?

### やりたいこと

BonsaiLog アプリには **「お手本の絵」** (mockup、 デザイナーさんが描いた完成イメージ) があります。
実機 (実際の Android スマホ) で動かしたアプリの画面が、 このお手本と「**ぴったり同じ**」 になるように直していく作業です。

### なぜやるの?

- ✅ ユーザー (盆栽愛好家のおじいちゃん / お父さん / 若手 IT 系の人) が見て **「あ、 きれい」 と思える品質** に
- ✅ お店 (App Store / Google Play) に出すための **基準を満たす**
- ✅ 19 言語対応で、 どの国の人が見ても **同じ品質**

### 今どこまで進んでる?

- 全 39 画面のうち、 **19 画面が「お手本と同じ ✅」 状態** (約 49%)
- 目標は **25 画面まで** (= MVP、 ストア出すための最低ライン)
- ゴールは **39 画面全部 100%**

### 例え話

| アプリ開発                   | 家のリフォーム                        |
| ---------------------------- | ------------------------------------- |
| **mockup** (お手本)          | リフォーム後の理想の写真              |
| **実機 SS** (今のスマホ画面) | 今の部屋の写真                        |
| **比較**                     | 「お手本と今」 を並べて違い探し       |
| **修正候補**                 | 「ここの壁紙、 こう変えますか?」 提案 |

- 👤 **あなた (user)** = 家の持ち主、 リフォーム発注者

## 3. 8 step ループ — 1 つずつ詳しく

### Step 1: 議論開始の予告 ⏱ 1 分

Claude が **「これから何件質問する、 何ラウンド議論する」** を最初に宣言します。

**家リフォームでいうと**: 「お客様、 今日の相談は **質問 3 件、 議論 2 ラウンド** くらいです」

**user 確認**: ❌ 不要 (議論開始の宣言のみ)

### Step 2: 比較レポート (お手本 vs 今) ⏱ 5 分

Claude が以下を **並列で読み込み**、 違いを表にします:

1. お手本 (mockup PNG)
2. 今のスマホ画面 (実機 SS)
3. 今のコード (RN ファイル)
4. 業務台帳 (skip-list.json / SCREEN_PAIRS)
5. 設計図 (関連 ADR、 例: ADR-0018)

**家リフォームでいうと**: 「お手本写真とお宅の今の写真、 並べて見ました。 7 個違うところを見つけました、 表にしますね」

**user 確認**: ❌ 不要 (Step 3 で承認求める)

### Step 3: 修正候補 + ペルソナ評価 + 質問 ⏱ 5 分

違いを「**許容できる差分**」 (環境差 / OS 差 / 既知 bug) と「**修正候補**」 に分けます。
修正候補ごとに **4 人の目線** (お年寄り / 若手 / プロ / 初心者) で評価し、 Claude の推薦と反対意見を提示。

**家リフォームでいうと**: 「7 個の違いのうち、 3 個は許容範囲です。 残り 4 個を直す候補にしました。 4 人の目線で評価しました、 全員にとって OK か確認済 ✕ なしです。 私の推薦は『候補 1』、 反対意見は『候補 2 はコスト高い』。 どれにしますか?」

**user 確認**: ⭐ **必須** (「Q-N1 a で OK」 等の短い回答)

### Step 4: 修正実行 ⏱ 5 分

user の OK 後、 関連画面も含めて **自動的に判定** してコード修正。
**ホットリロード** (5 秒以内に実機反映) で動作確認。

**家リフォームでいうと**: 「お客様 OK、 工事開始です。 ついでに同じ素材使ってる隣の部屋も一緒に直しますね」

**user 確認**: ❌ 不要 (Step 6 で再確認)

### Step 5: 自動カメラマンが撮影 ⏱ 3 分

**Maestro** という自動カメラマンが、 毎回同じ手順でアプリを操作 → 画面を撮影。
複数ページ画面 (例: 言語選択の 19 言語) は、 4 回撮影 + ImageMagick で縦に連結 = 1 枚の縦長画像。

**家リフォームでいうと**: 「工事完了写真撮りました。 リビング広いので、 部屋全体を写すために 4 回に分けて撮って、 縦長 1 枚に繋げました」

**user 確認**: ❌ 不要

### Step 6: 4 項目チェック + 業務台帳更新 ⏱ 3 分

新しい写真を **Claude 自身が** 見て、 4 項目チェック:

1. **タブ構成** (お手本と同じ?)
2. **セクション構成** (お手本と同じ?)
3. **UI 種別** (リスト or グリッド or タイムライン?)
4. **スクロール範囲** (全部スクロールできる?)

OK なら業務台帳 (skip-list.json) 更新。

**家リフォームでいうと**: 「お手本と完成写真、 4 つの基本項目で確認しました: 部屋の広さ ✅、 窓の位置 ✅、 ドア ✅、 収納 ✅。 業務日報も更新します」

**user 確認**: ❌ 不要

### Step 7: 完成アルバム更新 + Claude 自己検証 ⏱ 2 分

pairing-report (完成アルバム) を更新後、 **Claude が自分で「ちゃんと貼られた?」 と確認**。
過去にここを怠って user に二度手間させた事故あり (R-1 違反)、 構造的に防ぐ仕組み化済。

**家リフォームでいうと**: 「完成アルバム更新しました。 念のため、 私 (大工) 自身もアルバム開いて『新写真ちゃんと貼られてる?』 確認しました」

**user 確認**: ❌ 不要 (Claude 自己検証)

### Step 8: user 確認 + 工事完了 ⏱ 10-15 分

user に pairing-report URL + F5 (再読み込み) 依頼。
user OK 後:

1. 新しい作業帳 (`git checkout -b`) を作る
2. 変更ファイルを提出箱に入れる (`git add`)
3. ラベル貼り (`git commit`)
4. GitHub に送る (`git push`)
5. PR 作成 (`gh pr create`)
6. 自動検査員 CI が 4 種類チェック (`gh pr checks --watch`)
7. 本帳簿に登録 (`gh pr merge --squash`)
8. 工事日誌に記録 (Engram session_summary)

**user 確認**: ⭐ **必須** (pairing-report 目視 + 「OK」)

## 4. コマンドの意味 (初心者向け)

### 🐚 シェル

| コマンド             | 意味                 | 例え話                       |
| -------------------- | -------------------- | ---------------------------- |

| `grep '単語' <file>` | 単語検索             | 「本の中で『盆栽』 を探す」  |
| `mkdir -p <path>`    | フォルダ作成         | 「新しい部屋を作る」         |

### 🌳 git

| コマンド                      | 意味                   | 例え話                     |

| `git status`                  | 状況確認               | 「机に何の書類があるか」   |

## 5. user (あなた) がすること

### セッション開始時の prompt

**短縮版** (慣れた user 用):

```
[画面 id] を進めてください、 docs/how-to/ui-diff/screen-integration-loop.md の手順で。
```

**詳細版** (初めての user 用):

```
[画面 id 例: bonsai-tab] を、 mockup と実機を比べて整合性確認してください。
進め方は docs/how-to/ui-diff/screen-integration-loop.md の 8 step ループで。
- R-13 予告 (質問数 + ラウンド数) を冒頭で
- 修正候補は 4 ペルソナ評価 + 私の推薦 + 反対意見セットで
- 関連画面 (隣の画面で同じ修正が必要なら) 自動判定して一緒に直す
- 私の確認は 3 回以内に
```

→ `docs/how-to/workflow/prompts/screen-integration-prompt.md` に詳細な雛形あり

### user の確認ポイント (3 回以内)

1. **Step 3**: 「修正候補のうち、 どれを採用するか」 を回答 (「Q-N1 a で OK」 等)
2. **Step 8 前半**: 「pairing-report を F5 で見て」 → 「OK」 or 「もう少し直して」
3. **Step 8 後半 (任意)**: commit 前の最終確認 (省略可)

### 拒否権

- 「**stop**」 「**待って**」 で即停止可能 (kill-switch flag も別途用意)
- 「scope 大きすぎ」 と感じたら「修正は 3 件以内に絞って」 等で scope 縮小指示可

## 6. うまくいかない時

### 症状別 対処

| 症状                                  | 対処                                                                                           |
| ------------------------------------- | ---------------------------------------------------------------------------------------------- |
| Maestro flow が失敗 (PASS にならない) | `capture-with-logs.sh` の summary.md + logcat.log で原因切り分け                               |
| ホットリロードが反映しない            | Native コード変更なら rebuild 必要 (`pnpm build:android:dev:local`)、 JS 変更なら Metro 再起動 |
| pairing-report で新 SS が反映されない | skip-list の `reevalArtifact` path 確認、 capture-app.sh 新形式 = `<artifact>/app/<id>.png`    |
| R-25 構造的差分が新たに見つかった     | 修正候補に追加、 user 判定 (修正 or 受容)                                                      |
| CI fail                               | `gh pr checks` で失敗 job 確認、 ローカル `pnpm verify` 再実行で再現確認                       |

### よくある誤解

- ❌ 「mockup と実機が **画素単位** で完全一致するべき」 → ✅ R-25 構造系 4 項目で「**意味的に同じ**」 なら整合性 lv2 達成、 ピクセル単位の完全一致は不要
- ❌ 「全 39 画面で同じ手順使う必要」 → ✅ **目的優先、 手順は guideline**、 画面特性 (Splash / OS ダイアログ等) で柔軟判断
- ❌ 「整合済 = 修正不要」 → ✅ 既 achieved でも user 感覚で再確認 OK、 改善余地ある場合は新ループへ

## 9. まとめ

- **目的**: お手本 (mockup) と実機を比べて、 同じになるように直す
- **手順**: 8 step ループ (技術者向け詳細は `screen-integration-loop.md`)
- **道具**: Claude (大工) / Maestro (自動カメラマン) / ImageMagick (写真連結) / GitHub (本帳簿)
- **あなたの役割**: 「[画面 id] を進めて」 と指示するだけ、 確認は 3 回以内
- **満足条件**: pairing-report.html で「お手本 ⇔ 実機」 が同じに見える状態

**「ガチガチに手順守る必要なし、 目的 (= 同じに見える) が達成できれば OK」**。 これが BonsaiLog の UI 整合ループです。
