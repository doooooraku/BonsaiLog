## 2-3. ユーザーの変化（Before→After）

### 高橋さん62歳（父から25本継承）の場合

- **Before**: Notion自作リマインダーが日付集中し1日で25本は不可能。父の毛筆帳面はシミで読めず、今年の植替え時期を1本ミスして根詰まりで枯死。市場価値¥50,000〜¥200,000損失。「親父の代から大事にしてた真柏を、俺の代で枯らしちまった」
- **After**: 鉢の前でワンタップ記録。樹種ごとに作業タイミング分散通知。父の記録が年次タイムラインに再構成され自分の記録と接続。**「引き継いだ25本を、俺の代で1本も失わずに、息子に渡せる」**

### Marcus 35歳（米国ポートランド・屋内6本）の場合

- **Before**: Excelに全木のID、親株、pH値、針金日まで記録。屋外で開くのが面倒で記録が途切れ、ジュニパーに針金食い込みで永久傷。Instagram投稿用の成長写真が日付順にならない。
- **After**: 写真とメモが樹木ごとに自動整理。年次タイムラインがインスタ映え縦長画像で出力可能。針金日から外し日が自動計算。**"I can finally say I'm a bonsai parent, not a plant murderer."**

## 3-2. 機能の詳細説明（小中学生にもわかる言葉）

### 🔧 樹種別初期ガイドカード（🩹1対応）

**やること**: 「新しく黒松を追加」とタップすると、アプリが**30日分のto-doカレンダー**を自動で作る。

- 1-3日目: 「置き場所を決める（日光が当たる屋外）」
- 5-7日目: 「水やり状態を毎日確認」
- 10日目: 「葉の色をチェック」
- 30日目: 「1ヶ月経過。枯れていなければ生着」

**なぜ効く？**: 新規盆栽の75%が1ヶ月で枯れる現実。**最初の30日が勝負**。カレンダーがあれば「何をすればいいか分からない」が消える。

### 🔧 水やり履歴グラフ（🩹2対応）

**やること**: 水やりボタンをタップした日が**棒グラフ**で見える。最後の水やりから何日経ったかがアプリ目立つ場所に表示される。

**重要な設計**: **「水が足りません」と判定しない**。ただ事実を見せるだけ。ユーザーが気づく。医療アプリ的な診断責任を回避する重要原則。

**コード例**（開発者向け、小中学生向け解説付き）:

```dart
// 「最後の水やりから何日経ったか」を計算する関数
int daysSinceLastWatering(DateTime lastWatered) {
  // DateTime.now() = 今この瞬間の日時
  // difference() = 2つの日時の差を計算
  // inDays = その差を「日数」で取り出す
  return DateTime.now().difference(lastWatered).inDays;
}

// 画面に表示する文字を作る関数
String displayText(DateTime lastWatered) {
  final days = daysSinceLastWatering(lastWatered);
  if (days == 0) return "今日、水やりしました";
  return "最後の水やりから$days日";  // 判定しない、事実だけ
}
```

**コマンド解説**:

- `DateTime`: 日時を扱う「型」。2026年4月23日10時 のような情報を入れる箱
- `.difference()`: 2つの日時を引き算して「違い」を出す
- `.inDays`: その違いを「日数だけ」で取り出す。「3日と5時間」なら `3` が返る

### 🔧 分散アルゴリズム（🩹3対応）

**やること**: 25本の樹の作業が**同じ日に集中しないように**ずらす。

**仕組み**:

1. 各樹の「次の作業予定日」を計算
2. 同じ週に3件以上集中していたら、前後1週間にずらす
3. ユーザーには「推奨作業日: 4/28〜5/5」のように**幅**で提案

```dart
// 推奨作業日を週単位で分散させる関数
DateTime suggestWorkDate(Tree tree, List<Tree> allTrees) {
  final ideal = tree.nextIdealWorkDate();  // 理想的な作業日

  // その週に既に予定されている他の樹の数をカウント
  final sameWeekCount = allTrees
      .where((t) => t.id != tree.id)
      .where((t) => sameWeek(t.nextIdealWorkDate(), ideal))
      .length;

  // 3件以上なら前後にずらす
  if (sameWeekCount >= 3) {
    return ideal.add(Duration(days: 7));
  }
  return ideal;
}
```

**コマンド解説**:

- `List<Tree>`: Tree型を複数まとめた「リスト」（箱）
- `.where()`: 条件に合うものだけ抜き出す
- `Duration(days: 7)`: 7日間の時間の長さ

### 🔧 お引っ越し機能（🩹10対応）

**やること**: スマホ買い替え時、古いスマホから新しいスマホへ**ZIPファイル1個で**データ全部を移す。

**仕組み**（ADR-0007 で確定、F-11）:

1. 古いスマホで「バックアップを作成」をタップ
2. アプリが ZIP ファイル（manifest.json + 写真 + DB スナップショット）を生成
3. **Share Sheet**（標準 OS UI）が起動 → ユーザーが任意の保存先（iCloud Drive / Google Drive / メール / LINE 等）を選択
4. 新しいスマホで保存先からファイルを開き、BonsaiLog で「バックアップから復元」→ ファイル選択 → 復元

**設計方針**:

- **暗号化なし**（位置情報は USDA Hardiness Zone レベルのため PII リスク低、constraints §1-3）。UI で「クラウド保存時はユーザー責任」を明示
- **写真は長辺 2048 px にリサイズ**して ZIP に同梱（バックアップサイズ 200 MB ハード制限）
- **マージは追記のみ**（v1.0、ID 重複はスキップ）。完全置換モードは v1.1 以降検討
- **クロスプラットフォーム**（iOS ↔ Android 互換）

**なぜ必要？**: 高橋さん62歳の**「スマホ買い替えたらデータ消えるのでは」**が最大の不安。この解消で長期利用の決断ができる。標準 OS の Share Sheet UI は 9 年以上浸透した操作で、シニアにも分かりやすい。

> ⚠️ 当初設計（QR コード + WebRTC + AES-256 暗号化）から変更した理由は ADR-0007 を参照。実装コスト・テスト容易性・シニア UX・ADR-0005 (iOS 暗号化フラグ) との整合性を考慮した設計変更。

## 3-3. 継承モード（🩹7）の再定義

UXリサーチャー指摘「罪悪感は『予防できたはず』の後悔。過去を見せるだけでは解決しない」を受けて：

**→ 継承モードを「過去表示」から「世代を超える予防」に再定義**:

- 父の代の作業履歴を**学習データ**として提示（「去年のこの時期に父はこうしていた」）
- **同じ失敗を繰り返さないための情報**を前面に
- **Erikson の Generativity**（50-65歳の発達課題「次世代への貢献」）と一致する設計

# 5. 体験設計の原則（5つ、不変条件）

## 原則P1: 鉢の前で、ひと手入れ

- 屋外・汚れた手・片手操作を前提
- 主要アクションは **1タップ** で到達
- 大型ボタン、手袋でも操作可能なヒットエリア

## 原則P2: 診断しない、記録する

- AIで「この樹は病気です」と判定しない
- 示すのは「前回から X 日経過」「樹種別の一般的な目安」など**事実情報**のみ
- **PictureThis 15本全部誤同定事件の教訓**: AIに頼らないことが差別化
- 「参考」「目安」「テンプレート」の表現を徹底

## 原則P3: 永く、変わらない

- 流行色、過度なアニメーションを避ける
- 5年後も古く見えない伝統色（深緑・樹皮・和紙）
- 書体は可読性優先の明朝系＋セリフ系

## 原則P4: 控えめで美しく

- 可愛さ 5% に抑える（キャラを出しすぎない）
- 盆栽の「余白の美」を UI に反映
- 情報密度は低め、呼吸する空間

## 原則P5: 屋外で読める

- WCAG AA 以上のコントラスト比を全画面で維持
- ダークモード必須（自動切替）
- 屋外直射日光下テストを CI に組込

### 用語解説

- **WCAG**: Web Content Accessibility Guidelines、ウェブアクセシビリティの国際基準
- **CI**（Continuous Integration）: コード変更のたび自動でテストやチェックを走らせる仕組み

## 6-1. v1.x でやるコア機能（14項目）

7. **PDF エクスポート**（展示会出品記録・青色申告対応）
8. **「お引っ越し」機能**（端末買替時のフルバックアップ→復元）
9. **19言語ローカライゼーション**（v1.0で全19言語、ar 非対応）

## 9-2. Claude Code 翻訳運用スキーム（予算ゼロ前提）

### ワークフロー

1. **マスター日本語ファイル**（`ja.json`）を開発者が完成させる
2. **Claude Code にプロンプトテンプレートで翻訳依頼**（一言語ずつ）
3. **自動検証スクリプト**（Python）で機械チェック:
   - キーの数が一致
   - プレースホルダ `{days}` 等が維持
   - 音訳必須用語（bonsai, yamadori等）が残っている
4. **Pseudo-localization** で画面崩れを先出し
5. **Back-translation** で意味ズレチェック
6. **Visual Regression Testing**（Percy / Chromatic）でレイアウト崩れ検出
7. **アプリ内フィードバックボタン**でユーザー報告を集める

### 翻訳ファイル構造（JSON）

```json
{
  "common.app_name": "BonsaiLog",
  "common.ok": "OK",
  "tree.list.empty": "まだ盆栽が登録されていません",
  "tree.species.kuromatsu": "黒松",
  "work.watering": "水やり",
  "reminder.days_since_watering": "最後の水やりから{days}日"
}
```

**解説**:

- `"common.app_name"`: 「共通で使うアプリ名前」の棚札
- `"BonsaiLog"`: 棚の中身（実際の文字）
- `{days}`: ここに数字が入る「穴」。**翻訳で絶対に変えてはいけない**（変えるとバグる）

### Claude Code プロンプトテンプレート

```
あなたは盆栽ケアアプリの翻訳者です。以下のJSONを{対象言語}に翻訳してください。

【絶対ルール】
1. JSON構造・キー名は不変
2. {days}, {count}, {name} 等のプレースホルダは翻訳しない
3. 以下は音訳（ローマ字）で維持:
   bonsai, yamadori, mame, shohin, akadama, kokedama,
   niwaki, karikomi, nebari, jin, shari
4. 樹種の学名（Pinus thunbergii等）はラテン語のまま
5. スタイル名は日本語音訳で維持:
   Chokkan, Moyogi, Shakan, Kengai, Han-Kengai, Bunjin-gi
6. {対象言語}のネイティブが自然と感じる表現
7. UIボタンは短く、説明は丁寧に
8. 盆栽文化を尊重する表現

【入力】
{ja.jsonの中身}

【出力】
{対象言語}.json（JSON厳守、コメント・末尾カンマなし）
```

### 自動検証スクリプト（Python）

```python
# scripts/validate_i18n.py
import json
import re
import sys

def load_json(path):
    with open(path, encoding='utf-8') as f:
        return json.load(f)

def validate(master_path, target_path):
    master = load_json(master_path)
    target = load_json(target_path)
    errors = []

    # 1. キー数が一致するか
    if set(master.keys()) != set(target.keys()):
        missing = set(master.keys()) - set(target.keys())
        extra = set(target.keys()) - set(master.keys())
        if missing:
            errors.append(f"翻訳漏れ: {missing}")
        if extra:
            errors.append(f"余分なキー: {extra}")

    # 2. プレースホルダ {days} が維持されているか
    pattern = re.compile(r'\{[a-zA-Z_]+\}')
    for key in master:
        if key not in target:
            continue
        master_ph = set(pattern.findall(master[key]))
        target_ph = set(pattern.findall(target[key]))
        if master_ph != target_ph:
            errors.append(f"プレースホルダ不一致 [{key}]")

    # 3. 音訳必須用語チェック
    must_not_translate = ['bonsai', 'yamadori', 'shohin', 'mame',
                          'akadama', 'kokedama', 'niwaki']
    for key, value in target.items():
        for term in must_not_translate:
            if term in master.get(key, '').lower():
                if term not in value.lower():
                    errors.append(f"音訳消失 [{key}]: {term}")

    return errors

if __name__ == '__main__':
    errors = validate(sys.argv[1], sys.argv[2])
    if errors:
        print('\n'.join(errors))
        sys.exit(1)
    print('OK')
    sys.exit(0)
```

**コマンド解説**:

- `import json/re/sys`: それぞれの機能を使えるようにする呪文
- `set()`: 重複のない集合。2集合の差で「どちらにしかないキー」がわかる
- `re.compile(r'\{[a-zA-Z_]+\}')`: `{days}` パターンを探すレシピ
- `sys.exit(1)`: エラー時の「失敗」合図。CI が止まる

## 11-2. R1: 枯死起因の訴訟リスク

### 5 Whys

1. なぜ訴訟？ → 「大事な1本を枯らした」と責任追求
2. なぜ？ → アプリ表示を専門家推奨として過信
3. なぜ？ → UI表現が断定調
4. なぜ？ → 表現ルールが文書化されていない
5. なぜ？ → 開発者が意識していない

### 恒久策

- **許可語リスト**: 記録・履歴・整理・参考・テンプレート・目安
- **禁止語リスト**: 診断・判定・推奨・べき・危険・病気
- 利用規約に**英日併記の免責文**:
  > "This app provides reference information only and does not guarantee plant health. All care decisions are at your own judgment."
  > 「本アプリは参考情報を提供するもので、植物の健康を保証しません。すべての判断はご自身の責任で行ってください。」
- 樹種DB各エントリに**出典併記**（盆栽教本・協会資料）

## 11-3. R2: 19言語翻訳品質リスク

### 5 Whys

1. なぜ品質不足？ → Claude Code単独では文化ニュアンス外す
2. なぜ？ → ネイティブレビュー予算ゼロ
3. なぜ？ → コスト制約
4. なぜ？ → 個人開発

### 恒久策

- **Pseudo-localization + Visual Regression TestingをCIに必須組込**
- **アプリ内フィードバックボタン**全画面右上
- Reddit r/bonsai、r/Flutter/ReactNatve で**ベータテスター募集**（クレジット引き換え）
- 各言語App Storeレビュー月次チェック、「翻訳が変」3件で再翻訳
- ar は v1.x 全期間で非対応（RTL 回避）、印・泰は v1.0 から含む

## 11-4. R3: ストア審査での誤判定

### 恒久策

- マーケ露出で「reminder」「tracker」使用禁止、**「log / journal / record / 台帳 / 履歴」に統一**
- カテゴリを**ライフスタイル**に固定
- 審査前チェックリストをdocs/how-to/に組込
- **Google Play Disruptive Ads Policy違反回避**: Home下部単一バナーのみ、Interstitial禁止

## 11-5. R4: 市場獲得失敗

### 恒久策

- **kill criteria厳格運用**（§8-3）
- リリース前からReddit r/bonsai、Instagram #bonsai、YouTube Mirai Bonsai / Herons Bonsaiコメント欄でオーガニック認知
- 6ヶ月未達なら**ターゲット再定義**（植物全般拡大？）
- 12ヶ月未達なら**撤退orピボット**

## 11-6. R5: Bonsai Empire追随

### 恒久策

- **独自価値4点セット固定**: 19言語 / 完全ローカル / 買切 / 5年データ
- **5年分のユーザー独自DB** = 自然なベンダーロック
- **継承モード**（家族の代を接続） = Bonsai Empireには真似できない感情的価値
- **盆栽の家系図**（親木→子木の挿し木系譜） = オリジナル機能

## 11-7. R6: 広告フィッシング被害（新規）

### 5 Whys

1. なぜ被害？ → シニアが広告と本編を誤タップ
2. なぜ誤タップ？ → Xボタン小、広告領域が本編と近い
3. なぜ近い？ → Googleデフォルト使用
4. なぜ？ → 設計時に高齢者UX未検証
5. なぜ？ → 60代モニター不在

### 恒久策

- **60代モニター3人にプロトタイプ検証**（SNS無料公募）
- AdMob `setTargeting()` で**センシティブカテゴリ全拒否**
- 広告領域背景に**薄いグレー枠＋「広告」ラベル**
- **Xボタン48dp以上、周囲16dp余白**
- **週次でユーザー報告確認、問題広告即ブロック**

## 11-8. R7: Claude Code時代の同等アプリ量産（新規）

### 5 Whys

1. なぜ差別化が消える？ → LLMが同等機能を短時間で作る
2. なぜ作れる？ → コア機能が単位では真似可能
3. なぜ真似可能？ → 独自性が技術ではなく**知識と文化理解**にあるべきところ、ない
4. なぜ？ → 知識と文化で戦う設計になっていない

### 恒久策

- **知識の深さ**で差別化: 50樹種の作業タイミングDBに**盆栽師監修**を明記
- **盆栽文化理解**: 用語集19言語版を**世界唯一の公開資産**に
- **5年データ**: ユーザー長期蓄積データはLLMが即座に作れない

## 11-9. R8: プラットフォーム変化への追随（新規）

Bonsai Album（Andrew Nicolle、20年継続個人開発）が売上減に苦しんだ主因は**iPhone X以降の画面対応**（開発者ブログ2019-2021）。

### 恒久策

- **Flutter / React Native等クロスプラットフォーム採用**（iOS/Android変更コスト最小化）
- **年次予算5%をOSバージョン追随に確保**
- **サブスク収益の一部を保険積立**: プラットフォーム変化コストを吸収

## 付録B: 小中学生向け用語ミニ辞典

- **アダプティブバナー**: 画面サイズに合わせて広告が自動でちょうど良い大きさになる仕組み
- **ADR**: 「なぜこう決めたか」を残す小さな日記
- **API**: アプリ同士が情報をやり取りするための窓口
- **ASO**: App Store Optimization。アプリストアで見つけやすくする工夫
- **CI**: コードが変わるたび自動でテストやチェックを走らせる仕組み
- **CTR**: Click Through Rate。広告のクリック率
- **eCPM**: 広告1000回表示あたりの収益
- **Figma**: 世界中のデザイナーが使う画面デザインツール
- **i18n**: internationalizationの略（iと18文字とn）。多言語対応
- **JSON**: データを書く形式。人間もコンピュータも読める
- **kill criteria**: 撤退の判断基準
- **MoSCoW法**: 必要度の分類。Must/Should/Could/Won't
- **Pseudo-localization**: 翻訳前に「わざと長い変な文字」に置き換えて画面崩れを先出し
- **PR**: Pull Request。コードを本体に取り込む前のレビュー依頼
- **WCAG**: ウェブアクセシビリティの国際基準
