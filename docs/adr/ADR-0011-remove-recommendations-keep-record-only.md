## Context（背景：いま何に困っている？）

- 現状：
  - 戦略書 `product_strategy.md` L41 の BonsaiLog 定義は「**鉢 1 本ずつ一生分記録する**、オフライン完結の盆栽台帳アプリ」
  - L44 ポジショニング: 「**AI で判定するのではなく、ユーザーの判断を記録して次世代に残すためにある**」
  - `constraints.md` §1-4 AI 非搭載原則: 「アプリの立ち位置は『**台帳**（記録するもの）』であり、『**判定するもの**』ではない」
  - `constraints.md` §5-2 禁止語: 「診断 / 判定 / **推奨** / べき」を CI で検出して落とす
- 困りごと：
  1. **戦略・制約と F-03 機能の矛盾**: F-03「樹種別作業タイミング**推奨**」「30 日**ガイドカード**自動生成」は「記録のみ」哲学に反する
  2. **F-06 は F-03 の入力**: F-03 が削除されれば F-06 (地域気候・半球判定) も連鎖的に不要
  3. **法的リスク**: 推奨が外れた時のクレーム（樹を枯らした → アプリのせい）
  4. **過剰設計**: USDA Zone データ 3MB + species_tasks 780KB のバンドル肥大、シニア低速回線で離脱要因
  5. **ユーザー判断**: 「ユーザーは盆栽愛好家、すでに知識がある。提案は水を指す可能性」
  6. **競合差別化**: 11 痛みのうち 🩹1 (新規 75% 枯死) と 🩹4 (半球差) への対応が消えるが、Day One / Bear / Notion 等の「記録特化」業界標準と整合
- 制約/前提：
  - `constraints.md` §1-4 AI 非搭載原則
  - `constraints.md` §5-2 禁止語（診断 / 判定 / 推奨 / べき）
  - `product_strategy.md` L41 BonsaiLog 定義（記録）
  - 4 ペルソナ視点（高橋さん 62 歳経験者 / Marcus 35 歳 Excel 派 / 盆栽園プロ / ライトユーザー）

## Decision（決めたこと：結論）

### 完全削除する機能

1. **F-03 樹種別作業タイミング計算**（functional_spec §10、basic_spec §F-03）
2. **F-06 地域気候・半球対応**（functional_spec §11、basic_spec §F-06）
3. **30 日ガイドカード自動生成**
4. **F-05 自動分散アルゴリズム**（自動生成全廃、ユーザー手動設定のみ）
5. **species_tasks テーブル**（780KB）
6. **USDA Zone データ**（3MB）
7. **IANA timezone → 国 → 緯度経度ロジック**
8. **半球反転ロジック**
9. **F-13 Pro メリット「樹種別タイミング計算 Pro 限定」**（F-03 自体が消えるため）

### F-07 針金がけ記録の改修

10. ユーザーが針金がけイベント記録時に「**外す予定日時**」を任意入力できる
11. 指定日時に通知発火
12. 同時に「**装着期間 6 週経過しました**」事実通知も対応
13. **通知文言の方針**:

- ✅ 「針金の指定日時です（◯月◯日設定）」
- ✅ 「装着期間 6 週経過しました」
- ❌ 「針金を外しましょう」（推奨 / 命令、禁止）

### F-05 「気遣い型」ポップアップ（自動分散の代替）

14. **発火条件**: ユーザーが **6 件目の予定を登録しようとした時**（5 件超）
15. **ポップアップ文言**:
    ```
    「この日は既に 5 件の予定があります。
     無理のない範囲で進めてくださいね」
    [ そのまま登録 ] (デフォルト)
    [ 一覧を見る ]
    [ 今後表示しない ]
    ```
16. **設定**:
    - デフォルト ON
    - Settings → 通知設定 → 「予定が多い時の確認ポップアップ」トグルで OFF 可能
    - 盆栽園スタッフ等業務利用者向け抑制機能

### オンボーディング 5 ステップチュートリアル（新規、ADR-0014 で Step 5 追加）

17. **必須ステップ 5 つ**:
    - Step 1: 盆栽の登録（樹種 + 名前 + 写真）
    - Step 2: 作業の水やり記録（ワンタップ）
    - Step 3: タイムライン（予定 / 実績の見方、ADR-0008 status 列）
    - Step 4: **ヒートマップの読み方**（F-04、ADR-0013）
      - サンプルヒートマップ表示（実データではないデモ）
      - 凡例説明: `□ 0 回 ■ 1 回 ■ 2 回 ■ 3+ 回`
      - 「緑が濃いほど、その日に何回も水やりした日です」
      - 「マス目をタップすると、その日の詳細が見られます」
    - Step 5: **通知設定**（F-16、ADR-0014）
      - アイコン: 🔔 ベル
      - タイトル: 「通知で水やりを忘れない」
      - 本文: 「作業タイミングの目安を通知でお知らせします。通知はオフにもできます。いつでも設定から変更可能です。」
      - メインボタン: 「通知を有効にする」 → OS 通知許可リクエスト → 許可後 Step 5-B 水やり時刻設定 (ドロップダウン 1-5 回 + OS 標準タイムピッカー) → 「始める」で完了
      - サブリンク: 「あとで」 → 通知 OFF 状態で完了
      - OS 拒否時: 「通知を有効にしませんでした」トースト 1 秒 → 完了
18. **スキップボタン必須**（各 Step、Step 5 では「あとで」リンクで代替）
19. **スキップ時の通知初期値**: 通知マスタートグル OFF（ADR-0014 K1）。後で Settings で個別 ON 時のデフォルトは水やり 1 回 / 朝 07:00、当日まとめ朝 07:00。
20. Settings 「ヘルプ」→「チュートリアルを再表示」で再閲覧可能（許可済なら Step 5-A「通知をオン中です」+「水やり時刻を変更」ボタン）
21. 19 言語ローカライズ

### 戦略書ポジショニング再定義

21. **旧**: 「樹種×**地域気候**×作業履歴を、鉢 1 本ずつ一生分記録する、オフライン完結の盆栽台帳アプリ」
22. **新**: 「**樹種 × 作業履歴**を、鉢 1 本ずつ一生分記録する、オフライン完結の盆栽台帳アプリ」
23. **「地域気候」削除**、「判断はユーザー、記録はアプリ」哲学を貫徹

### 痛み 11 個の更新

24. 🩹1 新規 75% 枯死 → **対応削除**（戦略から削除）
25. 🩹3 リマインダー集中 → **「気遣い型」ポップアップに格下げ**
26. 🩹4 南北半球差 → **対応削除**（戦略から削除）
27. 🩹6 針金食い込み → **維持 + 改良**（指定日時通知）
28. 🩹2 / 🩹5 / 🩹7〜🩹11 → 維持（影響なし）

### F-04 水やり履歴グラフ

29. **判定なし、グラフ可視化のみ**（事実表示）

## Decision Drivers（判断の軸）

- Driver 1: **戦略書の「記録のみ」哲学の完全貫徹** — 仕様書内矛盾解消（推奨 vs 記録）
- Driver 2: **constraints §1-4 AI 非搭載原則 / §5-2 禁止語との完全整合**
- Driver 3: **法的リスク 0** — 推奨機能ゼロで責任回避
- Driver 4: **データバンドル 3.8MB 削減** — シニア低速回線で離脱要因削除（USDA 3MB + species_tasks 780KB）
- Driver 5: **4 ペルソナ全員での採用 OK** — 高橋さん経験者 / Marcus 派 / 盆栽園プロ / ライトユーザー
- Driver 6: **業界標準準拠** — Day One / Bear / Notion の「記録特化」路線、競合 Bonsai Care App の「南半球設定逆」問題の構造的回避
- Driver 7: **コスト 0** — ライセンス料 0、外部 API 0、実装/テスト工数大幅削減

## Alternatives considered（他の案と却下理由）

### Option A: 既存仕様維持（F-03 推奨機能 + F-06 USDA Zone）

- 概要: functional_spec §10 / §11 をそのまま実装、推奨機能を維持
- 良い点: 戦略書 🩹1 / 🩹4 への対応維持、競合差別化要素 11 個全部
- 悪い点: 戦略書「記録のみ」と矛盾、constraints §5-2 禁止語使用、法的リスク、データバンドル 3.8MB、シニア UX 不可解（USDA Zone 9a）
- 却下理由: 仕様書内矛盾、ユーザー判断で「水を指すから不要」と確定

### Option B: F-03 を「参考情報」に格下げ + F-06 簡略化（前回議論の C 案）

- 概要: 推奨を削除、樹種別参考情報の表示のみ、F-06 は国選択 + 半球判定のみ
- 良い点: 戦略整合、データバンドル削減、🩹1 への薄い対応維持
- 悪い点: 「参考情報」「国選択のみ」が活用しきれない（ユーザー指摘）、面倒な情報を残すリスク
- 却下理由: ユーザー判断「要らないものはユーザーにとって面倒な情報、入れたくない」

### Option C: F-03 / F-06 完全削除 + F-07 改良 + F-05 気遣い型 + チュートリアル（採用）

- 概要: 推奨機能ゼロ、F-07 はユーザー指定日時通知、F-05 は気遣い型ポップアップ、必須チュートリアル新規
- 良い点: 戦略整合完全、法的リスク 0、データバンドル 3.8MB 削減、4 ペルソナ全員採用 OK
- 悪い点: 🩹1 / 🩹4 への対応消失、競合差別化要素 2 個減
- 却下理由: なし（採用）

### Option D: F-03 / F-06 / F-05 / F-07 全削除（最小化）

- 概要: 推奨/通知関連を全廃、純粋な記録のみ
- 良い点: 哲学的に最も純粋
- 悪い点: F-07 針金がけ通知（ユーザー指定）まで失う → 🩹6 への対応消失、Marcus / 盆栽園で不満
- 却下理由: ユーザー判断「F-07 針金がけ通知はあってもいい」

## Consequences（結果）

### Positive（嬉しい）

- 戦略書「記録のみ」哲学を仕様書内で完全貫徹、矛盾解消
- constraints §1-4 / §5-2 と完全整合
- 法的リスク 0（推奨機能ゼロ）
- データバンドル 3.8MB 削減（シニア低速回線で離脱要因削除）
- 実装コスト大幅削減（USDA Zone ロジック / species_tasks / 半球反転 / 自動分散すべて削除）
- テストカバレッジ大幅削減（境界値テスト 15,600 パターン削減）
- 4 ペルソナ全員で採用 OK（盆栽園 OFF 設定、Marcus スキップ機能で対応）
- 業界標準（Day One / Bear / Notion）路線、ジャーナル / 記録系として正統
- ユーザーの主体性尊重（自分の判断を記録）

### Negative（辛い/副作用）

- **🩹1 / 🩹4 への対応消失**: 戦略書から痛み 2 個分の差別化要素が消える
- **ライトユーザー（新規購入者）への対応薄**: 「30 日ガイド」を期待する初心者層への配慮減
- **F-13 Pro メリット再構築**: 「樹種別タイミング計算 Pro 限定」削除により Pro 訴求が変わる
- **既存ドキュメント大規模書き換え**: basic_spec / functional_spec / product_strategy / constraints の 4 ファイル更新
- **F-05「気遣い型」ポップアップの心理学リスク**: 介入のし過ぎで離脱要因になる可能性 → デフォルト ON だが OFF 可能で吸収

## Rollout / Rollback（出し方/戻し方）

- リリース手順への影響：
  - F-03 / F-06 を v1.0 から削除して提出
  - F-07 改良 + F-05 気遣い型 + チュートリアルを v1.0 で実装
  - リリースノート: 「自分のペースで盆栽を記録できます」「針金作業時に外し日時を予約できます」
- ロールバック方針：
  - F-03 / F-06 復活は破壊的変更（spec 大規模追加）→ v1.0.x ホットフィックスでは行わない
  - F-07 改良 / F-05 気遣い型 / チュートリアルの個別無効化は UI 側でハードコード非表示で対応
- 検知方法：
  - ストアレビューで「ガイドが欲しい」「推奨機能がない」キーワード監視
  - DAU / Pro 加入率の v1.x モニタリング（ライトユーザー離脱が確認されたら v1.x で「初心者向けガイド (任意)」検討）
  - F-05 ポップアップの OFF 率モニタリング（盆栽園プロ層の離脱判断材料）

ADR-0009 で確定していた「樹種別タイミング計算 Pro 限定」が削除されるため、Pro メリットを再構築:

**旧 Pro メリット**（ADR-0009）:

- 写真の枚数を増やす
- **樹種ごとの作業時期を見る** ← 削除

- 写真の枚数を増やす（無制限）

### Repolog との差分

Repolog は推奨機能を持たないジャーナル系アプリ。BonsaiLog の F-03 / F-06 削除により Repolog 路線と同じ「記録特化」になる。Repolog 由来の lessons.md のパターン（記録 UX、相対パス保存、PRAGMA user_version）はそのまま適用。
