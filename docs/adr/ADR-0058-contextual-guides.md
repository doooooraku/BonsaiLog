# ADR-0058: 文脈内ガイド — スポットライト + Toast 流用で初回操作を導く (スライド型チュートリアルは不採用を維持)

- Status: Accepted
- Date: 2026-06-11
- Deciders: @doooooraku
- Related: Issue #1178 (基盤 + ガイド 5 種) / #1177 (空状態道しるべ化、merged) / #1179 (使い方ページ) / ADR-0018 (オンボ統合フロー) / ADR-0020 v1.x-2 (機能チュート撤去) / ADR-0014 Amended (通知 soft-ask) / ADR-0011 (記録のみ哲学)

---

## Context（背景：いま何に困っている？）

- 現状: ADR-0020 v1.x-2 (2026-05-16) で機能チュートリアル (tut1-4) を撤去した際、「機能発見性は Settings ヘルプで代替」としたが、その実体は通知画面 1 枚の再表示ボタンのみで、**「教える機能」を消した後の「発見させる機能」が未補填**だった (Sess102 なぜなぜ分析の根本原因)。
- 困りごと: 新規ユーザーが「最初の盆栽登録 → 最初の記録 → 予定」のゴールデンパスに自力で到達できるか、ユーザーの操作を導く仕掛けがない。
- 制約/前提:
  - ADR-0020 の撤去判断 (押し付けがましさゼロ / 完走率最大化) は**覆さない** — スライド型は今後も不採用
  - ADR-0011 記録のみ哲学: 文言は事実 + 操作案内のみ (推奨・診断 NG、`constraints.md` §5-2 禁止語)
  - 1 次情報: NN/g 70 名実験 (説明型チュートは成功率不変 + 主観難易度悪化) / Apple HIG・Material とも「説明より体験」/ 連続 coach mark は即 dismiss される (NN/g)
  - 表示形式 (スポットライト型) は user 提供のお手本スクショで合意 (Sess102)

## Decision（決めたこと：結論）

- 決定: **文脈内ガイド** を導入する — 教えたいことを「使う瞬間の画面の中」に置く。形式は (a) スポットライト (暗幕 + 本物の UI 要素の強調 + 吹き出し 1 つ) と (b) 既存 Toast の流用 (操作後のお祝い) の 2 種のみ。
- 適用範囲: v1.x、Free / Pro 共通。基盤 = `guidesStore` + `GuideSpotlight` + `guideTriggers` (#1178 PR-1)、配線は PR-2〜4。

### 設計原則 (5 箇条)

1. **1 画面 1 ガイド** — オーバーレイは同時に 1 つまで。連発した瞬間に読まれなくなる (NN/g)
2. **生涯 1 回** — `guidesStore.seen` (AsyncStorage 永続) で固定。再表示は設定 > 使い方の「画面のガイドをもう一度見る」(#1179、`resetAll`) 経由のみ
3. **必ず本物の UI 要素を指す** — ガイド専用ボタンは作らない。強調領域の tap は本物の操作を代行する (`onTargetPress`)。ユーザーが覚えるのは恒常導線そのもの (user 決定)
4. **全て dismiss 可能** — 暗幕 tap / あとで / OK のいずれでも閉じられ、閉じても二度と押し付けない
5. **ガイドを増やす時の第一候補は「使い方ページ行き」** — オーバーレイ追加は本 ADR の改訂を要する (ガイド疲れの構造防止)

### ガイド一覧 (5 種 + 欠番 1)

| ID                      | 発火条件 (すべて生涯 1 回)             | 型               | 強調対象                   |
| ----------------------- | -------------------------------------- | ---------------- | -------------------------- |
| g1RecordTabNudge        | Home focus 時に盆栽数 == 1             | スポットライト   | タブバー「記録」           |
| g2RecordCta             | 記録タブ初回 open                      | スポットライト   | BottomCtaBar「作業を記録」 |
| g3PlanCta               | 予定タブ初回 open                      | スポットライト   | BottomCtaBar「予定を追加」 |
| g5FirstRecordCelebrated | 作業記録の総件数 0→1 の保存直後        | Toast (既存流用) | — (お祝い文言のみ)         |
| g6DetailTabs            | 盆栽詳細初回 open                      | スポットライト   | タブ列 (作業履歴/作業予定) |
| ~~g4~~                  | **欠番** — 予定 1 件目のお祝いは不採用 | —                | —                          |

- g1 / g5 はデータ状態ガード付き (盆栽 2 本以上 / 記録ありの既存ユーザーには出さない)。g2 / g3 / g6 は既存ユーザーにも 1 回だけ出る (機能発見の救済を兼ねる)
- **g4 不採用の理由**: 予定 1 件目の登録直後は `optInPrompt` (ADR-0014 Amended の通知 soft-ask モーダル) が同瞬間に発火する。2 つの UI が同時に出るのは原則 1 違反のため、予定側のお祝いは作らない。○ 記号の説明は予定タブ空状態 (#1177) と使い方ページ (#1179) が担う

## Decision Drivers（判断の軸）

1. エビデンス準拠 — 説明型 (カルーセル / 連続 tips) は効果なし + 印象悪化が実験で確定している
2. ADR-0020 と無矛盾 — 撤去の決定を覆さず、未補填の代替を完成させる位置づけ
3. バンドル増ゼロ — スポットライトは Modal + 4 分割暗幕で自作、お祝いは既存 Toast 流用 (新規ライブラリなし、ADR-0018 踏襲)
4. i18n コスト最小 — オーバーレイ文言は短文 ×19 言語。ボタン名は直書きせず実ラベルキーを注入する (#1177 と同じ {cta} パターンに準拠)

## Options considered（検討した選択肢）

- **A. スライド式チュートリアル復活 (tut1-4 型)** — 不採用。NN/g 実験と ADR-0020 撤去理由に正面衝突
- **B. 連続ツアー (1/10 ステップ型、お手本スクショの原型)** — 不採用。形式のみ借用し、連続性 (次へ × N) は捨てる。3 ステップで完了率 72% → 7 ステップ 16% (Chameleon、ベンダー調査)
- **C. 文脈内ガイド (本決定)** — 採用
- **D. touch passthrough (Modal 越しに本物の要素を直接 tap させる)** — 不採用。platform 差異が大きく、`onTargetPress` による操作代行が決定的でテスト可能

## Consequences（結果と影響）

- 良い影響: ゴールデンパス (登録 → 記録 1 件 → 予定 1 件) の各分岐点に 1 つずつ道しるべが立つ。撤去済み「チュートリアル再表示」の嘘文言問題 (#1179) の受け皿が整う
- 注意点 (トレードオフ):
  - 対象座標の計測は `measureInWindow` 一択 (Fabric では measureLayout の relativeTo に数値ハンドル不可 — lessons 参照)。タブバー項目の計測が最難関のため g1 は PR-3 で PoC 先行、不可なら吹き出し位置を画面下固定に fallback
  - 初回セッションで最大 3 ガイド (g1→g2→g5) が連続しうる。各画面 1 つ・全て dismiss 可で許容と判断 (Sess102 議論)。これを超える追加は原則 5 でブロック
- テスト: `guidesStore` / `guideTriggers` / `spotlightLayout` は純関数 unit、E2E は golden path flow (PR-4 で追加)
